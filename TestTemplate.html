Week 1: APIs

What is an API?

API stands for Application Programming Interface.

For some context, APIs (Application Programming Interface) allow programmers, coders, or software engineers to add extra content before the real meat using previously-written code - usually by a company. For example, the Google Maps API allows a coder to use some of the functionality of Google Maps without having to re-code all of Google Maps on their own. Now, onto the real stuff.

By definition, an API is a set of programming instructions and standards to access software applications or Web tools. API interfaces define the method through which two software entities (or applications) communicate. To function, APIs make calls, which are requests of information, from various other applications (more on calls in a bit). Imagine APIs to be a middle-man between the coder and the application from which information is being taken. Let’s go back to our Google Maps example. If a programmer wanted to add a map on their webpage using the Google Maps API, he or she would send a request to the Google Maps API. The API would then approve of this request, contact the Google Maps API and return the map/information back the user.

Back to calls. We mentioned that APIs make calls, otherwise known as requests of information from various other applications. So how do calls work? Calls are managed by XML, a language through which programming applications communicate over the web. API XMLs allow users to make HTTP requests, which is basically a method of getting from one site to another on the internet - in layman's terms. There are four different HTTP methods that can be used when calling an API: GET, POST, PUT, DELETE. The endpoint of an HTTP is essentially the endpoint of the API; here is where an API’s data is retrieved. Hopefully that clarifies calling a bit!
